kubectl pod debugging

Change the variables below — every snippet on the page updates live. Click Explain under any command to see what each flag does, plus other flags worth knowing.

1. List pods and get status

List pods in the namespace with extra detail (node, IP, restart count):

bash
Explain

Lists every pod in namespace on context , using the wide output format so you also see node assignment, pod IP, nominated node, and readiness gates.

Flags used

--context=NAME
Name of the kubeconfig context to use for this command (global flag).
-n, --namespace=NAME
Target namespace for this request (global flag).
-o, --output=FORMAT
Output format. wide appends node and IP columns; other values include json, yaml, name, jsonpath, custom-columns.

Other flags you might want

-A, --all-namespaces
List pods across every namespace — ignores -n.
-l, --selector=KEY=VAL
Filter by label, e.g. -l app=web,tier=frontend.
--field-selector=KEY=VAL
Filter by field, e.g. --field-selector=status.phase=Pending.
-w, --watch
Stream changes as they happen instead of exiting.
--show-labels
Append a LABELS column to the default table output.
-L, --label-columns=LIST
Promote specific labels into their own columns.

Explain

Prints the full server-side object for pod as YAML. Inspect status.containerStatuses[] for restart counts and last-termination reasons, and status.conditions[] for scheduling/readiness gates.

-n, --namespace=NAME
Target namespace.

Explain

Prints the last log lines from pod . If the pod has multiple containers, kubectl uses the kubectl.kubernetes.io/default-container annotation, or errors and prompts for -c.

Lines of recent log file to display. Defaults to -1 (everything available).

Other flags you might want

Stream new log lines as they appear (like tail -f).

Logs from the previous container instance — essential after a crash.

Only logs newer than a relative time, e.g. --since=1h or --since=10m.

--all-containers
Logs from every container in the pod.
-l, --selector=KEY=VAL
Logs from every pod matching a label selector.

5. Copy files in and out

Pull a file from the pod down to your local machine. Requires tar to be present in the container image:

bash
Explain

Copies out of container in pod to local path . Under the hood kubectl streams a tar archive over an exec session, so the container image must contain a tar binary.

Flags used

--context=NAME
Kubeconfig context to use.
-n, --namespace=NAME
Target namespace.
-c, --container=NAME
Container to copy from / to. Defaults to the pod's default container.

Other flags you might want

--no-preserve
Do not preserve the copied file's ownership/permissions.
--retries=N
Retry the copy on failure. Negative values retry indefinitely; 0 disables retries (the default).

For symlinks, wildcards, or huge transfers, prefer kubectl exec pod -- tar cf - /path | tar xf - on the local side.

Push a local file into the pod (e.g. a config snippet or debug tool):

Explain

Uploads local file to inside container of pod . The target directory must already exist and be writable by the container's user.

Flags used

-n, --namespace=NAME

-c, --container=NAME
Destination container in the pod.

Other flags you might want

--no-preserve
Skip preserving file ownership/permissions inside the container.
--retries=N
Retry the transfer on failure (negative = infinite).

Path form {{namespace}}/{{pod}}:{{remote_file}} is also accepted — useful when you don't want to pass -n separately.

6. Port-forward

Forward a local port to a pod's port — open a private tunnel without exposing the pod via a Service. Leave it running in a separate terminal:

bash
Explain

Binds local port on 127.0.0.1 and forwards traffic to port inside pod . The session terminates if the selected pod dies — rerun the command to resume forwarding.

Flags used

--context=NAME
Kubeconfig context to use.
-n, --namespace=NAME
Target namespace.
LOCAL_PORT:REMOTE_PORT
Port pair. Use :REMOTE_PORT to let the OS pick a random local port.

Other flags you might want

--address=IP[,IP...]
Addresses to bind to. Defaults to localhost; use 0.0.0.0 to expose to the LAN (carefully).
--pod-running-timeout=DURATION
Wait up to this long for the pod to be running before failing.
deployment/NAME / service/NAME
Forward against a Deployment or Service instead of a pod; kubectl picks a backing pod.
NAMED_PORT
Use a Service's named port, e.g. 8443:https.

7. Events and resource usage

Recent events sorted by time — this surfaces scheduler decisions, image pulls, OOM kills, probe failures, and evictions across the namespace:

bash
Explain

Lists events in namespace ordered by their lastTimestamp field, so newest entries land at the bottom of the table. Events are retained for one hour by default, so check soon after the incident.

Flags used

--context=NAME
Kubeconfig context to use.
-n, --namespace=NAME
Target namespace.
--sort-by=JSONPATH
Sort the table by a field's JSONPath. .lastTimestamp gives chronological order.

Other flags you might want

kubectl events --for pod/NAME
The newer kubectl events subcommand can filter directly to one resource.
--field-selector type=Warning
Only show Warning events (drop the Normal noise).
-A, --all-namespaces
Show events across every namespace.
-w, --watch
Stream new events as they arrive.
-o yaml
Full event objects, including the involvedObject reference and counts.

8. Delete (force-restart)

Delete the pod. If it's owned by a Deployment / StatefulSet / ReplicaSet the controller will recreate it — effectively a forced restart. The default --cascade=background means dependents are GC'd asynchronously:

bash
Explain

Sends a graceful delete for pod . The pod receives SIGTERM, its terminationGracePeriodSeconds elapses, and the kubelet then issues SIGKILL. Use this — not --force — for almost every scenario; force should be reserved for unreachable nodes.

Flags used

--context=NAME
Kubeconfig context to use.
-n, --namespace=NAME
Target namespace.

Other flags you might want

--grace-period=N
Override the pod's termination grace period (seconds). 0 requires --force.
--now
Shorthand for grace-period 1 — quick but still graceful-ish.
--force
Immediately remove the API object without waiting for confirmation that processes have stopped. Risk of split-brain — use only on dead nodes.
--wait=false
Return immediately instead of blocking until the object is gone.
--dry-run=client|server
Show what would happen without actually deleting.
-l, --selector=KEY=VAL
Delete every pod matching a label selector.
