docker daily-driver

The Docker CLI commands you'll actually reach for on a working day — build, run, inspect, log, clean up. Change the variables below and every snippet updates live. Click Explain under any command for what each flag does, plus other flags worth knowing.

Variables

1. Build images

Build a tagged image from a Dockerfile. The final positional argument is the build context — the directory whose contents are sent to the builder and visible to COPY / ADD instructions:

bash
Explain

Builds an image from using as the build context, and tags the result as . On modern Docker this is actually a buildx build under the hood.

Flags used

-t, --tag stringArray
Image identifier in the form [registry/]repository[:tag]. Repeat for multiple tags.
-f, --file string
Name of the Dockerfile. Defaults to PATH/Dockerfile if omitted.
PATH | URL | -
Build context. A dot means the current directory; a URL points at a Git repo or tarball; - reads a Dockerfile from stdin.

Other flags you might want

--no-cache
Don't use any cached layers — every step re-runs.
--pull
Always attempt to pull a newer version of every referenced base image.
--build-arg KEY=VAL
Set a build-time variable consumed by ARG in the Dockerfile.
--target STAGE
Stop at a specific stage in a multi-stage build.
--platform os/arch
Build for a specific target platform, e.g. linux/amd64 or linux/arm64.
--secret id=mysecret,src=…
Expose a secret to RUN --mount=type=secret steps without baking it into a layer.
--push
Shorthand for --output=type=registry — push the built image straight to its registry.
--load
Shorthand for --output=type=docker — load the result into the local image store (needed when building with buildx for the local daemon).

Explain

3. Run an interactive shell

Spin up a throwaway container and drop into a shell — handy for poking at an image's filesystem, testing commands, or trying out a tool you don't want to install locally:

bash
Explain

Starts with a TTY attached and runs /bin/sh inside it. Exiting the shell stops the container; --rm then removes it. Try /bin/bash for Debian/Ubuntu-based images; /bin/sh is the safe default for Alpine. Distroless images may have no shell at all.

Flags used

-i, --interactive
Keep STDIN open even when not attached.
-t, --tty
Allocate a pseudo-TTY (combine with -i for a usable shell).
--rm
Remove the container automatically when it exits.

Other flags you might want

--entrypoint CMD
Override the image's ENTRYPOINT — useful when the default would prevent you reaching a shell.
-u, --user UID[:GID]
Run the shell as a specific user.
-v PATH:PATH
Mount a host path so files survive after the throwaway container is gone.
--platform os/arch
Pin the platform when running multi-arch images on Apple Silicon, etc.

7. Stop and remove

Graceful stop — sends SIGTERM, waits for the timeout (default 10s), then SIGKILL:

bash
Explain

Politely asks to stop. The container's PID 1 receives SIGTERM; if it's still running after the timeout it gets SIGKILL. Almost always what you want — reserve docker kill for stuck processes.

Flags used

(no flags)
Defaults: SIGTERM, 10-second timeout.

Other flags you might want

-t, --timeout N
Seconds to wait between SIGTERM and SIGKILL.
-s, --signal SIG
Send a different initial signal (e.g. SIGINT).

Remove a stopped container, including any anonymous volumes attached to it:

bash
Explain

Deletes and its anonymous volumes from the local daemon. Named volumes are unaffected — those are removed with docker volume rm.

Flags used

-v, --volumes
Remove anonymous volumes associated with the container.

Other flags you might want

-f, --force
Force removal of a running container — sends SIGKILL first.
-l, --link
Remove the specified container link, not the container itself (legacy networking).

Other flags you might want

9. Volumes and networks

Named volumes outlive containers — use them for any data you don't want to lose on docker rm:

bash
Explain

Lists, creates, and removes a named volume called . Mount it into a container with -v :/some/path.

Subcommands used

docker volume ls
List volumes.
docker volume create [NAME]
Create a named (or anonymous, if NAME omitted) volume using the default local driver.
docker volume rm NAME
Remove a volume. Fails if a container still mounts it.

Other subcommands you might want

docker volume inspect NAME
Show driver, mountpoint, labels, and creation time for a volume.
docker volume prune
Bulk-remove every volume nothing is using.
docker volume create -d DRIVER -o OPT=VAL
Use a non-default driver / pass driver-specific options.

User-defined bridge networks give containers DNS-based discovery by name — much nicer than the default bridge network:

bash
Explain

Lists, creates, and removes a user-defined bridge network called . Attach containers via --network ; they can then resolve each other by container name.

Subcommands used

docker network ls
List networks.
docker network create NAME
Create a network (defaults to the bridge driver).
docker network rm NAME
Remove a network. Fails if any container is still attached.

Other subcommands and flags you might want

docker network connect NAME CONTAINER
Attach a running container to an additional network.
docker network disconnect NAME CONTAINER
Detach a container from a network.
docker network inspect NAME
Show IPAM config, attached containers, options.
-d, --driver bridge|host|overlay|macvlan
Pick a non-default network driver on create.
--subnet CIDR
Override the auto-assigned subnet on create.
--internal
Restrict external access — container-only east-west traffic.
