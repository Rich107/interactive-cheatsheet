1. Basic GET with status code

bash
Explain

Performs a GET against and prints the HTTP status line plus response headers before the body.

Flags used

Other flags you might want

Same as -f, but still save the error body.

Ask the server for a compressed response and decompress on the fly.

2. Custom method and JSON body

POST a JSON payload with the correct content type:

bash
Explain

Sends a request to with a JSON body. When -d is present, curl defaults the method to POST — -X overrides that when you need PUT, PATCH, or DELETE.

Flags used

-X, --request METHOD
Override the HTTP method curl would otherwise pick.

5. Multipart file upload

Use -F to build a multipart/form-data body. Prefix a value with @ to attach a file; plain values become form fields:

bash
Explain

Uploads as the file field and attaches two text fields. curl picks a MIME boundary and sets Content-Type: multipart/form-data automatically — don't add a Content-Type header yourself.

Flags used

-F, --form NAME=CONTENT
Specify a multipart MIME field. Use @PATH to attach a file, <PATH to read text content from a file.
-X, --request METHOD
Override the HTTP method (curl defaults to POST when -F is present, so -X POST here is redundant but explicit).

Other flags you might want

--form-string NAME=VALUE
Like -F but never interprets @ or < as file references — useful when a value starts with one of those characters.
-F "file=@PATH;type=MIME"
Override the auto-detected content type for an attachment.
-F "file=@PATH;filename=NAME"
Force a different filename in the multipart part.
--form-escape
Backslash-escape special characters in field names and filenames.

10. TLS options

Use a custom CA bundle — required when the server cert is signed by an internal CA your system doesn't trust:

bash
Explain

Verifies the server certificate at against ./ca.pem instead of the system trust store.

Flags used

CA certificate (or bundle) to verify the peer against.

Other flags you might want

--capath DIR
Directory of CA certs (hash-named) for OpenSSL-style verification.

Use the operating system's certificate store explicitly.

Pin the server's public-key SHA-256 — defence-in-depth above CA validation.

Require an OCSP-stapled response from the server.
--crlfile FILE
Check the cert against this CRL.

Flags used

Passphrase for the private key.

Skip TLS verification entirely — useful against self-signed certs in dev:

bash
Explain
Warning: -k disables certificate validation entirely. The connection is encrypted but completely vulnerable to man-in-the-middle attacks — any attacker on the path can present their own cert and read/modify traffic. Never use -k against production hosts or with real credentials; restrict it to throwaway debugging against self-signed certs you control.

Allows curl to connect to even when the server certificate cannot be verified.

Flags used

-k, --insecure
Allow insecure server connections — skip CA validation, hostname check, and cert chain verification.

Safer alternatives

--cacert FILE
Trust the specific CA that issued the cert instead of disabling validation.
--resolve HOST:PORT:IP
Talk to a specific IP while keeping the original hostname for SNI/validation.
--ssl-no-revoke
Only skip CRL/OCSP checks (Schannel) — much narrower than -k.
--tls-max VERSION
Pin a TLS version to debug handshake issues without disabling validation.

11. Retries and timeouts

Retry transient failures and cap total transfer time — sensible defaults for any production script:

bash
Explain

Tries the request up to times with a 2-second pause between attempts, and aborts if the whole operation takes longer than seconds (including all retries). By default only transient transport errors retry — see --retry-all-errors to broaden it.

Flags used

--retry N
Retry the request if a transient problem occurs (DNS failure, refused connection, HTTP 408/429/500/502/503/504, etc.).
--retry-delay SECONDS
Fixed wait between retries (default uses exponential backoff).
-m, --max-time SECONDS
Maximum total time for the entire operation, including retries.

Other flags you might want

--retry-all-errors
Retry on any error, including HTTP 4xx — combine with --retry.
--retry-connrefused
Treat "connection refused" as transient (default: no).
--retry-max-time SECONDS
Stop retrying after this elapsed time, regardless of --retry.
--connect-timeout SECONDS
Maximum time allowed for the connection phase only.
--keepalive-time SECONDS
TCP keepalive probe interval.
