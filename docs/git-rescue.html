git rescue — undo & recover

Surgical recovery commands for when something goes wrong. Change the variables below — every snippet updates live. Click Explain under any command to see what each flag does. Flag text captured from git <subcmd> -h on git 2.50.1.

Variables

Explain

Prints the last 30 commits across every ref as a single ASCII graph, one line per commit, with branch/tag labels in the right column. Use this before any reset/rebase to confirm where you actually are.

Flags used

Other flags you might want

Annotate each commit with the ref it was reached through.

Follow only the first parent on merges — gives a linear view of mainline history.

Revert a merge commit (the -m 1 says "keep the first parent's line of history as the mainline"):

bash
Explain

For merge commits, git needs to know which parent represents "the branch you want to keep". -m 1 picks the first parent (usually the target branch you merged into). The diff of the second parent (the merged-in feature) gets inverted.

Reverting a merge makes future re-merges of the same branch not re-introduce the changes. To re-land later, revert the revert, or rebase the feature branch fresh on top.

Flags used

-m N
Select the Nth parent of a merge commit as the mainline.

10. Bisect to find a regression

Start the search, marking the current commit as bad and a known-good earlier commit:

bash
Explain

Initiates a binary search between (good) and the current commit (bad). Git checks out a midpoint commit; you test, then run git bisect good or git bisect bad and it picks the next one. Usually 5–10 iterations to isolate the breaker even across thousands of commits.

Subcommands used

bisect start
Begin a bisect session. Optionally takes <bad> <good>... inline.
bisect bad
Mark the current checkout as broken.
bisect good <rev>
Mark a revision as working.

Other subcommands you might want

bisect skip
Can't test this commit (broken build, missing dep) — git picks a neighbour.
bisect run <cmd>
Automate the search with a script that exits 0 (good) / non-zero (bad).
bisect terms
Show or rename the good/bad terms (e.g. fast/slow for performance bisects).
bisect log / bisect replay
Save and replay a bisect session.
bisect visualize
Open gitk showing the remaining suspect range.

Push it back with --force-with-lease — safer than --force because it aborts if someone else has since pushed:

bash
Explain
Force-push of any kind rewrites remote history. Coordinate with anyone who has branched off — they'll need to git fetch + git reset --hard {{remote}}/{{branch}} on their side. Never force-push to or other shared base branches.

Overwrites with your local . The --force-with-lease form checks that the remote ref is still at the SHA git last saw — if someone else has pushed in the meantime, the push aborts instead of clobbering their work.

Flags used

<remote>
The remote to push to (origin by default).
<refspec>
Branch to push.

Other flags you might want

-f, --force
Blunt force — no lease check. Easier to clobber colleagues' work. Avoid.
--force-if-includes
Tighter safety than --force-with-lease — also requires your local has integrated the latest remote updates.
-n, --dry-run
Show what would happen without actually pushing.
-u, --set-upstream
Set upstream tracking for the branch.
--no-verify
Bypass pre-push hooks. Use sparingly.
